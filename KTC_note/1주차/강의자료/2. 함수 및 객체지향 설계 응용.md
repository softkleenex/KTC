## 1️⃣ 함수 설계 — 읽기 좋은 함수 만들기

### 함수 분리 (Single Responsibility)

<aside> ✏️

하나의 함수가 너무 많은 일을 하면 수정할 때 어디서 버그가 났는지 찾기 어렵습니다. "한 함수 = 한 가지 역할" 원칙을 지키면 테스트하기 쉽고, 코드를 다시 봤을 때도 이해하기 쉽습니다.

</aside>

```python
# ❌ 나쁜 예: 한 함수가 계산+검증+결제를 몽땅 처리
def process_order(user, item, quantity):
    total = item["price"] * quantity       # 1. 계산
    if user["balance"] < total:            # 2. 검증
        return "잔액 부족"
    user["balance"] -= total               # 3. 결제
    return "주문 완료"

# ✅ 좋은 예: 핵심 기능을 부품처럼 쪼개고, 메인은 조립만 함
def get_total(item, quantity):
    return item["price"] * quantity

def has_enough_balance(user, amount):
    return user["balance"] >= amount

def pay(user, amount):
    user["balance"] -= amount

# [메인 함수] 쪼개진 함수들을 가져와서 '순서대로' 실행만 시킴
def process_order(user, item, quantity):
    total = get_total(item, quantity)
    
    if not has_enough_balance(user, total): 
        return "잔액 부족"
        
    pay(user, total)
    
    return "주문 완료"
```

### Early Return

<aside> ✏️

조건이 많아질수록 if 안에 if가 쌓이면서 코드가 복잡해 보일 수 있습니다. Early Return은 실패 조건을 먼저 쳐내고 핵심 로직에만 집중하는 패턴입니다. 로그인 체크, 입력 검증 등에서 활용하기 좋습니다.

</aside>

```python
# ❌ if 중첩 — 핵심 로직이 가장 안쪽에 숨어 있음
def process_order_nested(user, item, quantity):
    if user:
        if item:
            if quantity > 0:
                if user["balance"] >= item["price"] * quantity:
                    return "주문 완료"   # 읽으려면 4번 들여쓰기를 따라가야 함

# ✅ Early Return — 실패 조건을 먼저 처리
def process_order_early_return(user, item, quantity):
    if not user:
	    return "로그인이 필요합니다"
    
    if not item:
	    return "상품 정보가 없습니다"
        
    if quantity <= 0:
	    return "수량은 1 이상이어야 합니다"
        
    if user["balance"] < item["price"] * quantity:
	    return "잔액이 부족합니다"

    return "주문 완료"   # 여기까지 오면 모든 조건이 통과된 것
    
# --- 출력 결과 ---
    
print("=== Case 1: 정상 주문 (수량 2개 = 40,000원) ===")
print("중첩 IF 결과:", process_order_nested(user_profile, product_item, 2))
print("Early Return 결과:", process_order_early(user_profile, product_item, 2))
print("-" * 50)

print("=== Case 2: 실패 상황 (잔액 부족! 수량 3개 = 60,000원) ===")
print("중첩 IF 결과:", process_order_nested(user_profile, product_item, 3))
print("Early Return 결과:", process_order_early(user_profile, product_item, 3))
print("-" * 50)

print("=== Case 3: 실패 상황 (수량을 0개로 잘못 입력) ===")
print("중첩 IF 결과:", process_order_nested(user_profile, product_item, 0))
print("Early Return 결과:", process_order_early(user_profile, product_item, 0))
```

### 가변 인자 (_args,_ *kwargs)

<aside> ✏️

함수를 만들 때 인자 개수를 미리 모를 때가 있습니다. 예를 들어, AI API를 호출할 때 기본 옵션은 정해져 있지만 특정 상황에서만 추가 옵션을 붙여야 하는 경우가 있습니다. `*args`와 `**kwargs`는 이때 유용합니다.

> **비유:** 카페에서 주문할 때를 떠올려보면 이해가 쉽습니다. "아메리카노"는 기본 메뉴지만 어떤 손님은 "샷 추가, 얼음 적게, 시럽 빼주세요"라는 옵션을 붙일 수 있겠죠? 손님마다 옵션이 다르고 몇 개를 요청할지도 모릅니다. 이런 상황에서 유연하게 대처할 수 있도록 함수에서 구현한 것이 `**kwargs`입니다.

</aside>

#### `*args` vs `**kwargs` 비교

|**구분**|***args (가변 위치 인자)**|****kwargs (가변 키워드 인자)**|
|---|---|---|
|**의미**|**Arg**ument**s** (인자들)|**K**ey**w**ord **Args** (키워드 인자들)|
|**데이터 전달 방식**|**값만** 쉼표로 나열하여 전달||
|func("사과", "바나나")|**이름=값** 형태로 지정하여 전달func(model="gpt", temp=0.7)||
|**함수 내부 자료형**|**튜플 (Tuple)** 로 묶임||
|("사과", "바나나")|**딕셔너리 (Dictionary)** 로 묶임{"model": "gpt", "temp": 0.7}||
|**주요 사용 목적**|개수가 정해지지 않은 **여러 개의 단순 데이터**를 한 번에 받을 때|개수가 정해지지 않은 **추가 옵션/설정값**을 유연하게 받을 때|

```python
# *args — 개수가 정해지지 않은 위치 인자 (튜플로 받음)
def sum_all(*args):
    print(args)                 
    return sum(args)

print(sum_all(1, 2, 3))         # 6
print(sum_all(10, 20, 30, 40))  # 100
```

```python
# **kwargs — 이름 있는 추가 옵션 (딕셔너리로 받음)
def call_ai(prompt, **kwargs):
    options = {"model": "gpt-4", "temperature": 0.7}  # 기본값
    options.update(kwargs)                            # 추가 옵션으로 덮어씀
    print(f"프롬프트: {prompt}")
    print(f"옵션: {options}")

call_ai("안녕!")
# 프롬프트: 안녕!
# 옵션: {'model': 'gpt-4', 'temperature': 0.7}

call_ai("사진 분석", model="gpt-4o", temperature=0.2)
# 프롬프트: 사진 분석
# 옵션: {'model': 'gpt-4o', 'temperature': 0.2}
```

> 🚨 **순서 주의:** 함수 인자는 `일반 인자 → *args → **kwargs` 순서로 써야 한다.

> ex) `def func(name, *args, **kwargs)`

## 2️⃣ 객체지향 설계

### 상태와 책임 분리 → 객체지향 설계란?

<aside> ✏️

**1. 객체 = 레고 블록**

예전 방식(절차지향)은 레시피대로 요리를 하는 거라면, 객체지향은 **"스스로 움직이는 블록"**들을 만드는 거예요.

- 예를 들어 게임을 만든다면, '자동차' 블록, '운전자' 블록, '도로' 블록을 각각 따로 만듭니다.
- 각 블록 안에는 자기 데이터(색깔, 속도)와 할 일(달리기, 멈추기)이 다 들어있어요.

**2. 왜 이렇게 할까?**

- **고치기 쉽다:** 자동차 바퀴가 고장 나면 자동차 블록만 새로 끼우면 돼요. 다른 블록(운전자, 도로)은 건드릴 필요가 없죠.
- **다시 쓰기 좋다:** 한 번 잘 만든 '자동차' 블록은 다른 게임을 만들 때도 그대로 가져다 쓸 수 있어요.
- **확장이 쉽다:** '자동차' 블록을 기본으로 해서 날개만 달면 '비행기' 블록을 금방 만들 수 있어요. </aside>

```python
class ChatSession:
    def __init__(self, user_name):
        self.user_name = user_name     # 상태: 이 객체만의 데이터
        self.messages = []

    def add_message(self, role, content):   # 책임: 이 객체가 할 수 있는 일
        self.messages.append({"role": role, "content": content})

    def get_summary(self):
        return f"{self.user_name}의 대화 ({len(self.messages)}개)"

# 각 유저는 완전히 독립된 세션을 가진다
session_a = ChatSession("철수")
session_b = ChatSession("영희")

session_a.add_message("user", "안녕!")
print(session_a.get_summary())   # 철수의 대화 (1개)
print(session_b.get_summary())   # 영희의 대화 (0개) — 서로 영향 없음
```

- 상태: 객체가 기억하고 있는 것
- 책임: 객체가 스스로 수행하는 기능

### Data Class

<aside> ✏️

데이터를 저장하는 용도의 클래스를 간결하고 가독성 있게 정의하게 해주는 표준 라이브러리입니다.

데이터를 담는 클래스를 만들 때마다 `__init__`에 속성을 하나하나 쓰는 건 번거롭겠죠? `@dataclass`를 쓰면 초기화·출력·비교 메서드가 자동으로 생성됩니다. 복잡한 기능(메서드)보다는 이름, 나이, 이메일 같은 **정보를 보관하는 게 주 목적인 클래스**를 만들 때 사용합니다.

[https://docs.python.org/ko/3.14/library/dataclasses.html](https://docs.python.org/ko/3.14/library/dataclasses.html)

</aside>

#### 먼저 알아야 할 내용 - 자주 보이는 매직 메서드 정리

|**메서드명**|역할|목적|
|---|---|---|
|**init**|**객체 초기화**|객체 생성 시 필요한 초기 데이터(속성) 설정|
|**repr**|**디버깅용 출력**|개발자가 객체 상태를 파악하기 쉽게 출력|
|**eq**|**동등 비교**|객체 내부의 값을 기준으로 같은지 판단|
|**str**|**사용자용 출력**|사용자가 보기 쉬운 문자열|

- 매직 메서드: 파이썬의 클래스 안에 정의할 수 있는 특별한 메서드, 개발자가 직접 함수 호출하지 않아도 특정 상황에서 자동으로 호출

```python
from dataclasses import dataclass, field
from typing import List

# 기존 방식 — 반복적인 코드가 많음
class UserOld:
    def __init__(self, name: str, age: int, email: str):
        self.name = name
        self.age = age
        self.email = email

    def __repr__(self):
        # !r을 사용하면 문자열에 자동으로 따옴표(')가 붙어 출력됩니다.
        return f"UserOld(name={self.name!r}, age={self.age})"

u1 = UserOld("카카오", 23, "k@k.com")
u2 = UserOld("카카오", 23, "k@k.com")

print(u1)        # 출력: UserOld(name='카카오', age=23) -> __repr__ 덕분에 이렇게 나옴
print(u1 == u2)  # 출력: False -> __eq__를 안 만들었기 때문에 메모리 주소로 비교함

print("-" * 50)

# @dataclass 방식 — __init__, __repr__, __eq__ 자동 생성
@dataclass
class User:
    name: str
    age: int
    email: str
    tags: List[str] = field(default_factory=list)  # 리스트 기본값은 반드시 field() 사용
    
    # 일반 클래스에서는 tags = [] 로 쓰면 모든 인스턴스가 리스트를 공유하는 치명적 버그가 생깁니다.
    # @dataclass는 이를 방지하기 위해 tags = [] 라고 쓰면 아예 ValueError를 발생시킵니다.
    # 따라서 "인스턴스마다 새 리스트를 만들어라"라는 의미로 field(default_factory=list)를 써야 합니다.

user1 = User(name="카카오", age=23, email="kakao@kakao.com")
user2 = User(name="카카오", age=23, email="kakao@kakao.com")

print(user1)           # 출력: User(name='카카오', age=23, email='kakao@kakao.com', tags=[])
print(user1 == user2)  # 출력: True -> 값이 같으면 같은 객체로 판단 (__eq__ 자동 구현)
```

### 캡슐화

<aside> ✏️

캡슐화란 **데이터(상태)와 그 데이터를 다루는 기능(메서드)을 하나로 묶고, 외부에서 내부의 데이터를 함부로 들여다보거나 수정하지 못하게 막는 것**을 말합니다. 마치 알약의 캡슐이 내부 약가루를 보호하고 있는 것과 같습니다.

만약 은행 앱에서 유저가 자신의 계좌 잔액 숫자를 직접 수정할 수 있다면 어떻게 될까요? 신뢰를 잃고 아무도 그 앱을 사용하지 않겠죠. 그렇기 때문에 데이터는 숨기고, 미리 정해진 검증된 통로(입금 / 출금 메서드 등)을 통해서만 접근하게 만들어야 합니다.

</aside>

```python
class BankAccount:
    def __init__(self, owner):
        self.owner = owner
        self._balance = 0           # _ : "직접 접근 금지" 약속

    # 1. 검증된 통로 (입금)
    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("입금액은 0보다 커야 합니다")
        self._balance += amount

    # 2. 검증된 통로 (출금)
    def withdraw(self, amount):
        if amount > self._balance:
            raise ValueError("잔액 부족")
        self._balance -= amount

    # 3. 안전하게 읽기만 허용 (@property)
    @property
    def balance(self):
        return self._balance

account = BankAccount("카카오")
account.deposit(10000)

# ✅ 안전한 방식 (검증 로직 통과)
account.withdraw(3000)

print(f"현재 잔액: {account.balance}원")

# ❌ 위험한 방식 (직접 수정 시도)
# account.balance = 1000000    # 에러 발생! (@property라 수정 불가)
# account._balance = 1000000   # 가능은 하지만, 추천하지 않는 코드
```

챗봇 서비스를 만들었는데, 사용자가 늘어 서비스가 커진 상황을 가정해볼게요. "일반 유저 챗봇"과 "프리미엄 챗봇"처럼 비슷하지만 다른 버전이나 기능들이 필요해질 수 있습니다. 상속을 쓰면 공통 기능은 한 번만 작성하고, 달라지는 부분만 추가할 수 있습니다. 코드 중복 없이 기능을 확장할 수 있다는 게 장점입니다.

- ❌ ChatBot에 번역 기능이 필요하다 → ChatBot이 번역기는 아니잖아요?

단순히 기능을 빌려 쓰고 싶다면 상속보다 [**조합(Composition)**](https://www.notion.so/3642bb9842578079bd85f2fe3f172c84?pvs=21) 이 더 안전한 경우가 많아요.

#### **2단계: 기능 수정 (Overriding)**

부모의 기능은 마음에 안 들어서 내 식대로 '덮어쓰기' 하는 경우입니다.

```python
# 부모 클래스: 기본형 챗봇
class BasicBot:
    def say(self, msg):
        return f"🤖 [기본 응답]: {msg}"

# 자식 클래스: 부모의 특정 기능을 내 방식대로 '덮어쓴' 챗봇
class OverrideBot(BasicBot):
    # 부모와 '완전히 똑같은 이름'의 함수를 만들면 덮어쓰기(오버라이딩)가 됩니다.
    def say(self, msg):
        return f"✨ [덮어쓴 응답]: {msg}"

# 1. 기본형 봇에게 말을 걸었을 때
basic = BasicBot()
print("1. 기본형 봇의 결과:")
print(basic.say("안녕하세요"))

print("-" * 60)

# 2. 덮어쓰기형 봇에게 똑같이 말을 걸었을 때
override = OverrideBot()
print("2. 덮어쓰기형 봇의 결과:")
print(override.say("안녕하세요, 저는 님을 도우러 온 사람입니다."))  # 똑같은 say를 호출했지만 출력 결과가 다름
```

#### **3단계: 부모 기능 활용하기**

부모가 한 일에 내 할 일을 살짝 '덧붙이는' 경우입니다.

```python
class BasicBot:
    def say(self, msg):
        # 부모가 담당하는 '복잡하고 귀찮은' 작업들
        print("[시스템] DB에 메시지 기록 중...")
        print("[시스템] 욕설 필터링 검사 중...")

class PremiumBot(BasicBot):
    def say(self, msg):
        # 1. 부모가 하던 일을 먼저 시킴(super)
        res = super().say(msg)
        
        # 2. 부모가 일처리를 끝내고 준 결과물(res)에 내 것만 살짝 얹음
        return f"⭐[VIP] {res}"

bot = PremiumBot()
print(bot.say("오늘 하루 고생하셨습니다"))
```

#### OOP의 나머지 두 특징 — 다형성(Polymorphism)과 추상화(Abstraction)

**다형성 (Polymorphism)**

같은 메서드 이름인데, 클래스마다 다르게 동작하는 개념이에요.

비유: 리모컨의 "전원 버튼"은 TV에선 TV를 켜고, 에어컨에선 에어컨을 켜요. 이름은 같지만 대상에 따라 동작이 달라지죠.

```python
class KakaoBot:
    def respond(self, msg):
        return f"카카오: {msg}에 답변드립니다"

class ClovaBot:
    def respond(self, msg):
        return f"클로바: {msg}를 분석했어요"

# 어떤 봇인지 몰라도 respond()를 호출하면 각자 알아서 동작
bots = [KakaoBot(), ClovaBot()]
for bot in bots:
    print(bot.respond("날씨 알려줘"))
# 카카오: 날씨 알려줘에 답변드립니다
# 클로바: 날씨 알려줘를 분석했어요
```

---

**추상화 (Abstraction)**

내부 구현은 몰라도 되고, 어떻게 쓰는지만 알면 된다는 개념이에요.

비유: 자동차 핸들을 돌리면 바퀴가 꺾이는데, 그 안에서 어떤 부품들이 움직이는지 운전자가 알 필요는 없죠.

```python
from abc import ABC, abstractmethod

class BaseBot(ABC):
    @abstractmethod
    def respond(self, msg: str) -> str:
        """이 메서드는 반드시 구현해야 해요"""
        pass

class KakaoBot(BaseBot):
    def respond(self, msg):
        return f"카카오 답변: {msg}"

# BaseBot()  → 직접 생성하면 에러 (추상 클래스는 직접 쓸 수 없음)
# KakaoBot() → respond()를 구현했으니 OK
```

#### 조합 (Composition) — 기능이 필요하면 속성으로 가지기

```python
# ChatBot에 번역 기능이 필요하다면? ChatBot이 번역기를 '가지도록' 만들어요
class Translator:
    def translate(self, text):
        return f"[EN] {text}"  # 실제론 번역 API 호출

class ChatBot:
    def __init__(self, name):
        self.name = name
        self.translator = Translator()  # 상속 대신 속성으로 포함

    def respond(self, msg):
        return f"{self.name}: {msg}에 대한 답변"

    def respond_in_english(self, msg):
        response = self.respond(msg)
        return self.translator.translate(response)

bot = ChatBot("카카오봇")
print(bot.respond("안녕"))
# 카카오봇: 안녕에 대한 답변

print(bot.respond_in_english("안녕"))
# [EN] 카카오봇: 안녕에 대한 답변
```

> 이렇게 하면 Translator 기능이 바뀌어도 ChatBot 코드를 건드릴 필요가 없어요.

**실습 문제 — CafeMenu 클래스 완성하기**

```python
from dataclasses import dataclass

@dataclass
class CafeMenu:
    name: str
    price: int
    
    # [미션 1] 외부에서 직접 수정할 수 없는 '내부용 재고 변수'를 만드세요.(캡슐화)
    # ✏️ [미션 1] 내부용 재고 변수를 선언하세요 (변수명 앞에 _ 를 붙여 캡슐화)
    # 형식: _변수명: int = 10  (예: _stock: int = 10)

    # ✏️ [미션 2] @property 데코레이터를 사용해 외부 읽기 전용 속성을 만드세요
    # 이 메서드 위에 @property를 추가하세요
    def stock(self):
        return  # 여기에 내부 재고 변수를 반환하세요

    # ✏️ [미션 3] Early Return 패턴을 사용하여 주문 함수를 완성하세요
    def order(self, quantity=1):
        # ✏️ [미션 3-1] 수량이 0 이하일 때 "수량 오류"를 반환하는 조건문을 작성하세요
        if ...:  # 여기에 조건식을 작성하세요
            return "수량 오류"
            
        # ✏️ [미션 3-2] 재고가 부족할 때 "재고 부족 (현재 n개)"를 반환하는 조건문을 작성하세요
        if ...:  # 여기에 조건식을 작성하세요
            return f"재고 부족 (현재 {self._stock}개)"
        
        # ✏️ [미션 3-3] 재고를 차감하는 코드를 작성하세요
        # 여기에 재고 차감 코드를 작성하세요
        return f"{self.name} {quantity}잔 주문 완료"

# --- 아래는 코드 검증을 위한 테스트 구문입니다 ---
coffee = CafeMenu("아메리카노", 4500)

print(coffee.order(2))    # 아메리카노 2잔 주문 완료
print(f"남은 재고: {coffee.stock}") # 확인 가능

print(coffee.order(0))    # 수량 오류
print(coffee.order(15))   # 재고 부족 (현재 8개)

# 직접 수정 시도 (에러가 나야 캡슐화 성공!)
try:
    coffee.stock = 100
except AttributeError:
    print("성공: 재고를 직접 수정할 수 없습니다.")
```

- 힌트
    
    - 미션 1: `_stock: int = 10` — 변수명 앞에 `_`를 붙이면 "외부 접근 금지" 관례
    - 미션 2: `def stock(self)` 위에 `@property` 추가, `return self._stock`
    - 미션 3-1: `if quantity <= 0:` / 미션 3-2: `if self._stock < quantity:`
    - 미션 3-3: `self._stock -= quantity`
- 정답
    
    ```python
    from dataclasses import dataclass
    
    @dataclass
    class CafeMenu:
        name: str
        price: int
        _stock: int = 10  # 내부용 재고 변수 (캡슐화)
    
        @property
        def stock(self):
            return self._stock
    
        def order(self, quantity=1):
            if quantity <= 0:
                return "수량 오류"
            if self._stock < quantity:
                return f"재고 부족 (현재 {self._stock}개)"
            self._stock -= quantity
            return f"{self.name} {quantity}잔 주문 완료"
    
    coffee = CafeMenu("아메리카노", 4500)
    print(coffee.order(2))           # 아메리카노 2잔 주문 완료
    print(f"남은 재고: {coffee.stock}")  # 남은 재고: 8
    print(coffee.order(0))           # 수량 오류
    print(coffee.order(15))          # 재고 부족 (현재 8개)
    try:
        coffee.stock = 100
    except AttributeError:
        print("성공: 재고를 직접 수정할 수 없습니다.")
    ```
