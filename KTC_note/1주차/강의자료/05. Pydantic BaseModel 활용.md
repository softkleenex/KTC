
## 🤔 (복습) Pydantic 이란?

<aside> 📌

Python 타입 힌트를 이용해 **데이터를 자동으로 검증**해주는 라이브러리

- 잘못된 타입이 들어오면 자동으로 에러를 발생시켜줘요
- FastAPI와 함께 쓰여요 (FastAPI 내부에서 Pydantic을 씀)
- `if isinstance(age, int)` 같은 유효성 검사 코드를 직접 짜지 않아도 돼요. 타입만 선언하면 자동으로 검증해줘요 </aside>

<aside> 🏦

**비유**

Pydantic = **은행 창구 직원**

고객이 서류를 내밀면 "나이란에는 숫자만 써야죠", "이메일 형식이 틀렸어요"처럼 접수 전에 먼저 검토해줘요

</aside>

## 📥 요청 스키마 정의

<aside> 📌

- 클라이언트가 보내는 데이터의 **규격(틀)**을 정의해요
- 규격에 맞지 않으면 FastAPI가 자동으로 `422 Unprocessable Entity` 에러를 반환해요
- 별도의 if문 없이도 **자동으로 타입을 변환**해줘요 (예: `"1"` → `1`) </aside>

**기본 BaseModel**

```python
uvicorn 04_BaseModel:app --reload
```

```python
from pydantic import BaseModel
from typing import Optional

class MessageRequest(BaseModel):
    user_id: int
    content: str
    language: Optional[str] = "ko"  # 선택 필드, 기본값 "ko"

# ✅ 올바른 요청
# {"user_id": 1, "content": "안녕하세요"}
# → MessageRequest(user_id=1, content='안녕하세요', language='ko')

# ✅ 문자열 "1"도 자동으로 int 1로 변환
# {"user_id": "1", "content": "안녕"}
# → MessageRequest(user_id=1, content='안녕', language='ko')

# ❌ 타입 변환 불가능하면 에러
# {"user_id": "abc", "content": "안녕"}
# → 422 Unprocessable Entity
```

**Field로 세부 조건 추가**

<aside> 💡

**Field 옵션 정리**

- `min_length` / `max_length`: 문자열 길이
- `ge` (greater or equal): 이상 / `le` (less or equal): 이하
- `gt` / `lt`: 초과 / 미만
- `default`: 기본값 </aside>

```python
from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI()

class MessageRequest(BaseModel):
    user_id: int
    content: str = Field(min_length=1, max_length=5)   # 1~5자
    max_tokens: int = Field(default=500, ge=1, le=2000)  # 1~2000 사이만 허용

@app.post("/message")
def create_message(request: MessageRequest):
    return request
```

## 📤 응답 스키마 정의

<aside> 📌

- 서버가 반환하는 데이터의 **규격**도 정의할 수 있어요
- `response_model`로 지정하면 Swagger UI에 응답 형식이 자동 문서화돼요
- **내부 필드(비밀번호 등)를 응답에서 자동으로 제외**할 수 있어요 </aside>

```python
from fastapi import FastAPI
from pydantic import BaseModel, Field
from datetime import datetime

app = FastAPI()

# 클라이언트가 보내는 요청 데이터 구조 (비밀번호 포함)
class MessageRequest(BaseModel):
    user_id: int
    content: str = Field(min_length=1, max_length=5)
    password: str

# 서버가 밖으로 내보내는 응답 데이터 구조
class MessageResponse(BaseModel):
    id: int
    user_id: int
    content: str
    created_at: datetime
    status: str
    # password 필드는 없음 → response_model에 의해 자동으로 클라이언트에게 숨겨짐

@app.post("/messages", response_model=MessageResponse)
def create_message(request: MessageRequest):
    return MessageResponse(
        id=1,
        user_id=request.user_id,
        content=request.content,
        created_at=datetime.now(),
        status="sent",
    )
```

## ✅ 유효성 검증 심화

### 이메일, 문자열 길이 검증

<aside> 📦

EmailStr 사용 전 추가 설치 필요: `pip install pydantic[email]` — 설치 없이 임포트하면 `ImportError`가 발생합니다.

</aside>

**사용하는 이유?**

- **자동 검증:** 정규표현식을 직접 코딩하지 않아도 이메일 형식 오류를 API 진입 단계에서 자동 차단
- **코드 가독성:** `str` 대신 `EmailStr`을 명시함으로써 해당 필드에 이메일 데이터가 들어온다는 목적을 직관적으로 전
- **에러 메시지 표준화:** 형식이 맞지 않을 때 Pydantic이 전 세계 표준 규칙에 맞는 에러 응답(`value is not a valid email address`)을 알아서 반환

```jsx
python -m venv venv

# (Mac 사용자용 활성화)
source venv/bin/activate

# (Windows 사용자용 활성화 - CMD/PowerShell 공통)
.\\venv\\Scripts\\activate

pip install fastapi uvicorn "pydantic[email]"
```

```python
from pydantic import BaseModel, EmailStr, Field

class UserCreate(BaseModel):
    name: str = Field(min_length=2, max_length=20)
    email: EmailStr       # 이메일 형식 자동 검증
    age: int = Field(ge=0, le=150)
    bio: str = Field(default="", max_length=500)

class UserResponse(BaseModel):
    id: int
    name: str
    email: EmailStr
    age: int
    bio: str

@app.post("/users", response_model=UserResponse)
def create_user(user: UserCreate):
    return UserResponse(
        id=1,
        name=user.name,
        email=user.email,
        age=user.age,
        bio=user.bio,
    )

# ✅ 올바른 요청
# {"name": "카카오", "email": "kakao@kakao.com", "age": 23}
# → UserCreate(name='카카오', email='kakao@kakao.com', age=23, bio='')

# ❌ 에러 케이스들
# {"name": "a", ...}          → name은 최소 2자 이상
# {"email": "kakao-at-..."}   → 이메일 형식 불일치
# {"age": -1}                 → 0 이상이어야 함
```

### 커스텀 유효성 검사 (@field_validator) _(참고)_

<aside> ⏱️

`Field()`의 min_length, max_length, ge/le 옵션으로 대부분의 검증이 가능합니다. 그보다 복잡한 로직이 필요할 때 사용하며, 시간이 남을 때 확인 / 실습해보시는 걸 추천드립니다.

</aside>

```python
from pydantic import BaseModel, field_validator

class MessageRequest(BaseModel):
    content: str

    @field_validator("content")
    @classmethod
    def content_must_not_be_empty(cls, v):
        if not v.strip():
            raise ValueError("메시지 내용이 비어있을 수 없어요")
        if len(v) > 500:
            raise ValueError("메시지는 500자 이하여야 해요")
        return v.strip()  # 앞뒤 공백 자동 제거

# MessageRequest(content="  ")
# → ValueError: 메시지 내용이 비어있을 수 없어요

# MessageRequest(content="  안녕  ")
# → MessageRequest(content='안녕')  # 공백 자동 제거!
```

## 🏗️ Pydantic 모델 상속과 코드 재사용

<aside> 📌

**개념**

Pydantic 모델을 **상속**해서 공통 필드를 재사용하고, 상황에 맞게 확장할 수 있어요

</aside>

- 사용자가 보낸 요청 메시지를 검증하고, 서버가 처리 결과를 응답하며, VIP 전용 프리미엄 기능을 분기 처리하는 코드

```python
from pydantic import BaseModel, Field
from datetime import datetime

# 1. 공통 베이스 (부모 클래스)
class BaseMessage(BaseModel):
    content: str = Field(max_length=500)
    language: str = "ko"

# 2. 일반 유저의 요청 데이터 구조
class MessageRequest(BaseMessage):
    user_id: int
    
    """
    [최종 조립 결과]
    ⬇️ 부모 상속 : content (내용)
    ⬇️ 부모 상속 : language (언어 설정)
    ⭐️ 자식 추가 : user_id (보낸 유저 ID)
    """

# 3. 일반 유저에게 돌려줄 응답 데이터 구조
class MessageResponse(BaseMessage):
    id: int
    user_id: int
    created_at: datetime
    status: str

    """
    [최종 조립 결과]
    ⬇️ 부모 상속 : content (내용)
    ⬇️ 부모 상속 : language (언어 설정)
    ⭐️ 자식 추가 : id (메시지 고유 번호)
    ⭐️ 자식 추가 : user_id (보낸 유저 ID)
    ⭐️ 자식 추가 : created_at (전송 시간)
    ⭐️ 자식 추가 : status (전송 상태)
    """

# 4. VIP 유저 전용 프리미엄 요청 데이터 구조
class PremiumMessageRequest(MessageRequest):
    priority: int = Field(default=1, ge=1, le=3)

    """
    [최종 조립 결과]
    ⬇️ 조부모 상속 : content (내용)
    ⬇️ 조부모 상속 : language (언어 설정)
    ⬇️ 부모 상속   : user_id (보낸 유저 ID)
    ⭐️ 자식 추가   : priority (우선순위 1~3)
    """
```

<aside> 💡

**왜 이렇게 나누나요?**

- `MessageRequest`와 `MessageResponse`는 필드가 비슷해 보이지만 **역할이 다릅니다**
- 요청에는 없는 `id`, `created_at`이 응답에는 필요하고 비밀번호처럼 **요청엔 받지만 응답엔 숨겨야** 하는 필드도 있습니다
- 중복 코드 없이 기능을 **확장**할 수 있습니다 </aside>

## 📝 실습: **대학생 커뮤니티 서비스 게시글 API 설계**

<aside> 💡

**서비스 한 줄 소개**: 대학생들이 익명으로 소통하고 정보를 공유하는 커뮤니티 서비스입니다.

**기능 명세**: 사용자가 제목과 내용을 입력해 글을 올리면, 서버는 고유 번호(ID)와 작성 시간을 포함하여 응답합니다.

### 📋 세부 요구사항 및 문항별 힌트

**1. 공통 설계 (PostBase)**

- title: 문자열, 최소 2자 ~ 최대 50자 제한
    
- content: 문자열, 최소 5자 이상 입력 필수
    
- is_anonymous: 불리언(bool), 기본값은 True
    
- 💡 힌트
    
    `Field`, `min_length`, `max_length`, `default`
    

**2. 작성 요청 설계 (PostCreate)**

- PostBase를 상속받고, author_ip(문자열) 필드를 추가하세요.
    
- author_ip의 기본값은 "127.0.0.1”이어야 합니다.
    
- 💡 힌트
    
    `class PostCreate(PostBase):`
    

**3. 응답 결과 설계 (PostResponse)**

- PostBase를 상속받고, id(정수형)와 created_at(날짜시간) 필드를 추가하세요.
    
- 💡 힌트
    
    `int`, `datetime`
    

**4. API 엔드포인트 구현**

- @app.post("/posts") 경로를 생성하고, response_model을 설정하세요.
    
- 💡 힌트
    
    `response_model=PostResponse`
    

**5. 데이터 필터링 확인**

- Swagger UI에서 author_ip를 포함해 요청을 보냈을 때, 응답 결과에서 해당 필드가 자동으로 사라지는지 확인하세요. </aside>

    from pydantic import BaseModel, Field
    from datetime import datetime
    
    app = FastAPI()

    @app.post("/posts", response_model=PostResponse)
