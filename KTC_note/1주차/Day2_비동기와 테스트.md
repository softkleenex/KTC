



동기(앞 작업의 결과와 뒤 작업의 시작을 맞춘다)

비동기(앞 작업의 결과와 뒤 작업의 시작을 맞추지 않는다) 
	동시에 여러 연산을 하는것이 아니라, 대기 상태에 빠진 작업이 다른 작업에게 실행 흐름을 양보하는구조

동기와 비동기의 사전적 의미와 혼동하지말것


프로세스 : 실행중인 프로그램의 인스턴스
	독립적인 메모리

스레드 : 프로세스 내 실행작업의 단위
	메모리 공유
	하나의 프로세스 = 하나 이상의 스레드

파이썬에서는 비동기 프로그램 구성이 힘들다, GIL(Global Interpreter Lock가 그것을 막고 있기 때문이다.)

GIL은 한번에 하나의 스레드만 파이썬 코드를 실행하도록 제한하는 역할을 가진다.

GIL에 의해서 스레드를 아무리 늘려도 CPU연사ㄴ이 병렬로 빨라지지는 않는다. 하지만 비동기는 기다리는동안 권한을 넘기므로, 단일 프로세스 만으로 IO에서 큰 이득을 본다
(파이썬이 아니라 다른 작업은 파이썬이 기다리는동안 작업될 수 있으므로)

비동기 함수는 어떻게 만드는가…?

async + def로 만든다. 하지만 def로 만드는 일반함수와는 달리, 호출시에 바로 실행되는게 아니라 비동기적으로 실행될 수 있는 상태인 코루틴을 반환한다.


차이점1. 비동기 함수 안에서는 await을 사용할 수 있다
차이점2. 호출시에 실행되는게 아니라 코루틴 객체가 반환됨(class : coroutine)
	호출하고 실제로 실행하지않으면 RuntimeWarning 발행


그러면 어떻게 실행해…?

await에 대해서도 알아보자

이런식으로! coroutine 앞에다가 하면된다.
`await`은 **코루틴을 즉시 '실행'시키고, 그 작업이 끝날 때까지** 기다리는 동안 제어권을 이벤트 루프에 양보하는 것. (즉, 실행 버튼과 일시정지 버튼을 동시에 누르는 효과!)
일반함수에다가 사용시 syntaxerror발생하니 유의할것!


await asyncio.sleep(초)

일반 sleep은 모든 작업 종료, 하지만 asyncio sleep는 자신의 작업만 멈추고, 제어권을 이벤트 루프에 양보하는것.


실행하는건? asyncio.run(coroutine) = coroutine의 실행 결과값.

새 이벤트 루프 생성 > 코루틴 실행 > 루프 종료
프로그램에서 딱 한번만 호출!
이미 루프가 실행중일떄 asyncio.run()하면 RuntimeError 발생


즉, 비동기 환경을 생성하고 첫 작업을 구동하는 엔진의 시동 키 같은역할.

한번 시작한 작업이 필요없다면,,, 시스템 자원 확보를 위해서 즉시 중단해야함!!!


작업 중단은 Cancel() 사용

task.cancel()으로 취소 요청시에 await에서 CancelledError 발생함

try - except 보통 사용

| 에러                   | 발생 상황                          | 대처법                                  |
| :------------------- | :----------------------------- | :----------------------------------- |
| **`TimeoutError`**   | `wait_for()`에서 시간 초과           | `try/except`로 포착, 대체 로직 실행           |
| **`CancelledError`** | `task.cancel()`로 태스크 취소        | `try/except` 또는 `finally`에서 정리 작업    |
| **`RuntimeError`**   | 이미 루프 실행 중에 `asyncio.run()` 호출 | 중첩 호출 금지, `async` 함수 안에서는 `await` 사용 |
| **`ExceptionGroup`** | `TaskGroup` 내 여러 태스크에서 예외 발생   | `except*` 문법으로 타입별 분리 처리 (3.11+)     |
| **`RuntimeWarning`** | 코루틴을 `await` 없이 그냥 호출          | 반드시 `await` 또는 `create_task`로 실행     |

비동기에서 다루는 네트워크 클라이언트나 데이터 베이스 연결 객체같은건,,, 제대로 닫아줘야한다! 단순 변수가 아니라 물리적 자원이기 떄문에, 제대로 비동기로 열고 비동기로 닫아 주자.


async with 이라는 보호막 덕분에, 자원정리(Clean up)을 보장해준다.


| 상황 (Situation)  | 초보자의 접근 (Basic Async) | 실무자의 제어 (Managed Async) |
| :-------------- | :-------------------- | :---------------------- |
| 더 이상 필요 없는 작업   | 끝날 때까지 방치             | `cancel()` 호출 및 롤백      |
| 10,000개의 API 요청 | 한 번에 모두 실행            | `Semaphore`로 동시성 제한     |
| 네트워크 연결 사용      | 열어두고 방치함              | `async with`로 자동 정리     |
