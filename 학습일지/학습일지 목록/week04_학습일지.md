## 📚 이번 주 학습한 것

### 1. Next.js App Router와 컴포넌트 구조
- **핵심 개념**: App Router 기반 파일 라우팅, `page.tsx`, `layout.tsx`, 동적 라우팅(`[postId]`), Server Component와 Client Component의 역할 분리.
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - Next.js에서는 폴더와 파일 구조가 곧 URL 구조가 된다. React만 사용할 때는 라우팅, 데이터 페칭 방식, 폴더 구조를 직접 정해야 하지만, Next.js는 `app/posts/page.tsx`가 `/posts` 경로가 되는 식으로 프레임워크 차원의 규칙을 제공한다.
  - Server Component는 기본적으로 서버에서 실행되기 때문에 `process.env.FASTAPI_URL` 같은 서버 전용 환경변수를 안전하게 사용할 수 있다. 반대로 검색 입력, 삭제 확인창, 버튼 클릭처럼 브라우저 이벤트가 필요한 부분은 `"use client"`를 선언한 Client Component로 분리해야 한다.
  - 이번 주 실시간 강의에서 게시글 목록, 상세, 작성, 수정, 삭제 화면을 보며 서버 컴포넌트와 클라이언트 컴포넌트를 언제 나눠야 하는지 감을 잡을 수 있었다.

### 2. FastAPI와 Next.js를 연결한 풀스택 CRUD 흐름
- **핵심 개념**: FastAPI 백엔드의 `/posts` API와 Next.js 프론트엔드 화면을 연결하여 게시글 생성, 조회, 수정, 삭제를 처리하는 풀스택 구조.
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - FastAPI는 실제 데이터와 비즈니스 로직을 담당하고, Next.js는 사용자에게 보이는 화면과 사용자 입력 흐름을 담당한다. 프론트엔드와 백엔드가 분리되어 있어도 HTTP API라는 공통 인터페이스를 통해 연결된다.
  - 게시글 작성이나 수정처럼 서버 데이터가 바뀌는 작업은 Server Actions를 사용하면 별도의 API Route를 직접 만들지 않고도 서버에서 안전하게 처리할 수 있다. 처리 후 `revalidateTag()`로 캐시를 무효화하고 `redirect()`로 화면을 이동시키는 흐름이 실제 서비스의 CRUD 동작과 닮아 있었다.
  - 단순히 화면만 만드는 것이 아니라 DB에 데이터가 저장되고, 다시 목록과 상세 화면에 반영되는 흐름을 확인하면서 “풀스택”이라는 말이 프론트와 백엔드를 각각 만드는 것이 아니라 둘 사이의 데이터 흐름을 끝까지 이어주는 것이라는 점을 체감했다.

### 3. 검색 기능과 Route Handler 구조
- **핵심 개념**: Client Component에서 검색어를 상태로 관리하고, 전체 게시글 목록을 가져온 뒤 제목과 내용 기준으로 필터링하는 검색 기능. Direct Fetch 방식과 Route Handler 방식의 차이.
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - Direct Fetch는 브라우저가 FastAPI 주소를 직접 호출하는 방식이라 구조가 단순하지만, 백엔드 주소가 브라우저에 노출되고 CORS 설정이 필요하다. 이때 클라이언트에 노출되어도 되는 환경변수만 `NEXT_PUBLIC_` 접두사를 붙여 사용할 수 있다.
  - Route Handler 방식은 브라우저가 `/api/search` 같은 Next.js 내부 API를 호출하고, Next.js 서버가 FastAPI를 대신 호출하는 프록시 구조다. 브라우저 입장에서는 같은 Next.js 서버만 호출하므로 백엔드 주소 노출을 줄일 수 있고, CORS 문제도 완화된다.
  - `fetch`를 `axios`로 바꾸면서 `res.json()`을 직접 호출하지 않아도 되고, HTTP 에러가 자동으로 `catch`로 이동한다는 점이 편했다. 다만 라이브러리를 쓰더라도 결국 중요한 것은 요청 흐름과 에러 처리 구조를 이해하는 것이라고 느꼈다.

### 4. Vercel과 Railway를 이용한 배포 흐름
- **핵심 개념**: Next.js 프론트엔드는 Vercel에, FastAPI 백엔드는 Railway에 배포하고, 환경변수와 CORS 설정을 통해 두 서비스를 연결하는 과정.
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - 로컬에서는 `localhost:3000`과 `localhost:8000`이 내 컴퓨터 안에서만 통신하지만, 배포 후에는 Vercel과 Railway라는 서로 다른 서버가 공개 URL을 통해 통신한다. 그래서 프론트엔드 환경변수에는 로컬 주소가 아니라 Railway의 공개 URL을 넣어야 한다.
  - Vercel은 Next.js 배포에 최적화되어 있고 GitHub push 기반 자동 배포가 편리하다. 반면 FastAPI처럼 계속 실행되는 백엔드 서버와 SQLite 파일을 다루는 경우에는 Railway 같은 컨테이너 기반 호스팅이 더 적합하다.
  - 배포는 단순히 “사이트를 올리는 것”이 아니라, 환경변수, CORS, 빌드 로그, API URL, DB 저장 여부를 모두 맞춰야 완성되는 작업이라는 점이 인상적이었다.

## 🧱 막혔던 지점 & 해결 과정

### 1. 기말고사와 학습 진행 관리 부족으로 온라인 강의를 거의 듣지 못함
- **문제 상황**: 이번 주는 기말고사 일정과 개인 학습 진행 관리 부족이 겹치면서, 배정된 온라인 강의를 거의 듣지 못했다. 결국 전체 커리큘럼을 균형 있게 따라가기보다는 실시간 강의에서 다룬 내용 위주로만 학습하게 되었다.
- **시도한 방법**:
  * 실시간 강의는 최대한 놓치지 않으려고 했고, 강의 중 다룬 Next.js와 FastAPI 연동 흐름, 검색 기능, 배포 흐름을 중심으로 노트를 정리했다.
  * 온라인 강의를 다 듣지 못한 부분은 실습 코드와 강의자료를 보며 핵심 흐름만이라도 따라가려고 했다.
- **최종 해결 및 왜 그 해결책이 작동하는가**:
  * 이번 주 안에 모든 온라인 강의를 완벽히 따라잡지는 못했지만, 실시간 강의에서 직접 다룬 풀스택 프로젝트 흐름을 기준으로 최소한의 연결고리는 잡았다.
  * 특히 “FastAPI 백엔드 → Next.js 화면 → Route Handler → Vercel/Railway 배포”로 이어지는 큰 구조를 먼저 이해해두면, 이후 온라인 강의를 보충할 때 각 개념이 어디에 쓰이는지 더 잘 연결될 것 같다.

### 2. 로컬과 배포 환경의 차이를 한 번에 이해하기 어려웠음
- **문제 상황**: 로컬에서는 `localhost`만 신경 쓰면 됐지만, 배포에서는 Vercel URL, Railway URL, `FASTAPI_URL`, `NEXT_PUBLIC_` 환경변수, CORS 설정이 동시에 등장해 처음에는 흐름이 헷갈렸다.
- **시도한 방법**:
  * 브라우저가 직접 호출하는 값과 Next.js 서버에서만 읽는 값을 구분해서 정리했다.
  * Direct Fetch와 Route Handler를 표로 비교하며, 어떤 경우에 백엔드 주소가 브라우저에 노출되는지 확인했다.
- **최종 해결 및 왜 그 해결책이 작동하는가**:
  * `NEXT_PUBLIC_`은 브라우저에 공개되는 값, `FASTAPI_URL`은 서버 전용 값이라는 기준으로 나누니 훨씬 명확해졌다.
  * 배포 후 문제가 생기면 “프론트가 뜨는가 → 백엔드 `/docs`가 뜨는가 → 환경변수가 맞는가 → CORS가 맞는가 → 실제 CRUD가 되는가” 순서로 확인하면 된다는 점을 배웠다.

- **Try** 다음 주에 시도할 것: 기말고사 일정이 있더라도 최소 학습 시간을 미리 확보하고, 온라인 강의를 하루 단위로 쪼개서 밀리지 않게 듣기. 이미 놓친 4주차 온라인 강의는 주말에 한 번에 몰아보기보다, Day별로 나누어 실습 코드와 함께 다시 따라가며 부족한 부분을 보완하기.
