## 📚 이번 주 학습한 것

### 1. [Day 1] DOM 제어와 이벤트 모델 구조
- **핵심 개념**: DOM(Document Object Model)의 트리 구조, 요소 선택 및 동적 생성/추가, 이벤트 전파(Capturing, Target, Bubbling) 및 이벤트 위임
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - 브라우저가 HTML 코드를 파싱하여 메모리에 적재한 계층적 트리 구조가 DOM이다. 자바스크립트로 DOM을 조작하려면 `getElementById`나 `querySelector`로 노드를 탐색한 뒤 `createElement`와 `appendChild`로 물리적인 결합을 해줘야 비로소 화면에 렌더링된다.
  - 이벤트 발생 시 하위 노드에서 최상위 부모 노드 방향으로 이벤트가 전파되는 **이벤트 버블링(Bubbling)**은 브라우저의 기본 전파 메커니즘이다. 브라우저가 이 아키텍처를 선택한 이유는 수많은 자식 노드(예: 수백 개의 리스트 아이템)마다 리스너를 주입하는 성능 낭비를 막고, 공통 부모 요소에서 이벤트를 단 한 번만 감지하여 통합 처리하는 **이벤트 위임(Event Delegation)** 패턴을 지원하기 위함이다. 특정 자식의 액션으로 인해 의도치 않게 부모 이벤트 핸들러가 연쇄 반응하는 것을 방지하려면 `event.stopPropagation()`으로 버블링 단계를 끊어야 한다.

### 2. [Day 2] JS 엔진과 비동기 처리 런타임
- **핵심 개념**: ES6 문법(let/const 스코프, 구조 분해 할당, Spread), 고차 함수 및 배열 메서드(map, filter, reduce), 실행 컨텍스트, 이벤트 루프와 태스크 큐
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - `let`과 `const`는 함수 레벨 스코프를 가졌던 `var`와 달리 **블록 레벨 스코프**를 지원한다. 덕분에 제어문 블록 내부에서 선언된 변수가 전역으로 유출되는 오염을 원천 차단하고 스코프 체인을 통해 렉시컬 스코프 기반의 직관적인 상위 참조를 안전하게 돕는다.
  - 자바스크립트는 싱글 스레드 언어로, 연산을 처리하는 **콜 스택(Call Stack)**이 단 하나뿐이다. 그럼에도 불구하고 네트워크 요청이나 타이머 같은 비동기 작업이 블로킹 없이 처리되는 이유는 브라우저 런타임에 **태스크 큐(Task Queue)**와 **이벤트 루프(Event Loop)**가 존재하기 때문이다. 비동기 작업의 콜백 함수는 대기가 완료되면 태스크 큐에 적재되고, 이벤트 루프는 오직 **콜 스택이 완전히 비어있을 때**만 큐의 콜백을 스택으로 이동시켜 순차 실행한다. 이 메커니즘 때문에 `setTimeout(..., 0)`을 주더라도 현재 동기적으로 스택을 채우고 있는 전역 실행 컨텍스트가 종료될 때까지 콜백의 실행이 유예된다.

### 3. [Day 3] 비동기 데이터 통신(Promise)과 React 가상 DOM
- **핵심 개념**: Promise 상태(Pending, Fulfilled, Rejected)와 async/await, fetch 및 axios 통신, React 가상 DOM(Virtual DOM), JSX 문법과 useState/useEffect
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - 비동기 처리의 성공과 실패 결과를 약속(Promise) 객체로 감싸 구조화하고, 동기식 제어 흐름처럼 코드를 직관적으로 읽을 수 있도록 만든 문법이 `async/await`이다. 
  - 웹브라우저가 화면을 갱신하는 렌더링 파이프라인(Reflow/Repaint)은 연산 비용이 매우 무겁다. React는 이 성능 병목을 해결하기 위해 메모리상에 독립적으로 동작하는 **가상 DOM(Virtual DOM)** 트리를 둔다. 상태 변경이 일어나면 가상 DOM 트리끼리 메모리 상에서 비교(Reconciliation)하여 바뀐 조각들을 모으고(Batching), 최종 차이점만 실제 DOM에 '한 번에' 주입하는 효율적인 비교 알고리즘을 사용함으로써 화면 깜빡임과 렌더링 성능 저하를 방지한다.

### 4. [Day 4] React 컴포넌트 아키텍처와 스타일링
- **핵심 개념**: 단일 책임 원칙(SRP)에 따른 컴포넌트 분리, Props 단방향 흐름, Lifting State Up(상태 끌어올리기), Custom Hook, Tailwind CSS와 Flexbox
- **내가 이해한 방식 (왜 그렇게 동작하는가)**:
  - React는 의도적으로 데이터의 흐름을 **단방향(부모 ➔ 자식)**으로 제약한다. 양방향 데이터 바인딩 구조는 복잡한 상호작용 시 어떤 컴포넌트가 상태를 변조시켰는지 변화 경로를 역추적하기 극도로 어렵기 때문이다. 따라서 두 형제 컴포넌트가 데이터를 공유하기 위해선 공통 부모 컴포넌트로 상태의 소유권을 위임(`Lifting State Up`)하고, 자식은 단지 전달된 Props와 상태 변경 콜백 함수만 실행하여 예측 가능한 애플리케이션을 완성한다.
  - CSS 파일을 분리하지 않고 인라인 형태로 나열하는 **Tailwind CSS(Utility-First)**는 개발자가 클래스 네이밍 규칙에 얽매이는 수고를 없애고 HTML 컴포넌트 구조 내에서 스타일을 직관적으로 조립하게 돕는다. 빌드 타임에 사용된 유틸리티 클래스만 찾아 CSS 파일로 번들링하는 최적화 컴파일 아키텍처 덕분에 스타일 파일의 부피를 획기적으로 줄여 번들 용량을 감소시킨다.

## 🧱 막혔던 지점 & 해결 과정

### 1. useEffect의 의존성 배열 생략에 따른 API 호출 무한 루프 에러
- **문제 상황**: `useEffect` 내에서 서버 데이터를 비동기로 `fetch`하여 state(`setProducts`)를 업데이트하였더니, 브라우저가 먹통이 되며 API 호출이 무한 루프로 발생하는 현상이 일어남.
- **시도한 방법**: 비동기 호출 함수 자체의 순서를 바꾸거나, `fetch` 결과를 지역 변수에 직접 바인딩해보았으나 근본적인 현상은 그대로 유지됨.
- **최종 해결 및 왜 그 해결책이 작동하는가**:
  * `useEffect`에 **의존성 배열(Dependency Array)을 전달하지 않으면**, 컴포넌트가 렌더링될 때마다 이펙트가 실행됩니다. 이펙트 내부에서 `setProducts` 상태 업데이트 함수를 호출하면 강제로 리렌더링이 유발되며, 리렌더링으로 인해 또다시 `useEffect`가 실행되는 악순환이 발생했던 것입니다.
  * `useEffect` 호출 시 두 번째 인자로 빈 배열 `[]`을 전달하여 **최초 컴포넌트 마운트 시점에만 1회 호출**되도록 실행 타이밍을 통제하여 문제를 완벽히 해결했습니다.

### 2. 이벤트 위임(Event Delegation) 구현 시 자식 요소 클릭 타겟 오류
- **문제 상황**: 부모 리스트 `<ul>`에 이벤트 리스너를 한 번만 등록하고 자식 `<li>`에 저장된 `data-id` 값을 읽으려 했으나, 리스트 아이템 내부의 텍스트 영역(`<span>`)이나 프로필 이미지 태그(`<img>`)를 클릭하면 `id` 값을 정상적으로 읽지 못하고 `undefined`가 발생함.
- **시도한 방법**: `event.target.getAttribute("data-id")` 형식을 사용하였으나, 자식의 자식 노드인 `<span>` 등을 누르면 `data-id` 속성이 해당 태그에 없기 때문에 실패함.
- **최종 해결 및 왜 그 해결책이 작동하는가**:
  * `event.target`은 이벤트 버블링 단계의 기점인 **실제 클릭이 일어난 가장 하위의 구체적인 요소(Leaf Node)**를 가리킵니다. 반면 `event.currentTarget`은 이벤트 리스너가 실제로 부착된 요소(여기서는 `<ul>`)를 가리킵니다.
  * 리스트 아이템 단위의 정보(예: `<li>`에 부착된 데이터 속성)를 클릭된 위치와 무관하게 안전하게 취하기 위해, 자바스크립트의 `element.closest("li")` 메서드를 사용했습니다. 이를 통해 **이벤트가 시작된 노드(`target`)로부터 가장 가까운 부모 `<li>` 요소를 거슬러 올라가 탐색**함으로써, 자식 노드의 어떤 지점을 눌러도 의도한 데이터 속성을 유연하게 얻을 수 있도록 이벤트 위임 코드를 보강해 해결했습니다.

## 🔁 이번 주 회고 (KPT)
- **Keep** 유지하고 싶은 습관: 수업 진도를 따라 정리 노트를 작성할 때, 단순히 사용 문법을 복붙하기보다 브라우저 런타임 및 데이터 흐름 관점의 "왜(Why)"를 한 번 더 고민하고 문서화하는 습관.
- **Problem** 아쉬웠던 점: 금요일 즈음부터 엘리스 학습 사이트의 작동이 불안정하더니, 일요일에는 결국 사이트가 완전히 마비되어 3주차 강의를 끝까지 다 수강하지 못한 상태에서 제출 기한을 맞추기 위해 어쩔 수 없이 학습일지를 쓰게 되었다. 개인의 복습 부족이나 루틴 관리 미흡이 아니라, 학습 인프라(서버) 자체의 심각한 마비로 인해 강의 진도 수강 자체가 강제로 막혀버린 불가항력적인 상황이 발생하여 매우 답답하고 아쉽다. 다만, 객관적으로 돌이켜봤을 때 지금까지 계속 평일에 분량을 다 마치지 못하고 주말에 다 몰아서 강의를 마무리 짓는 경향성이 반복되고 있는 것 같아, 이 고질적인 학습 템포 자체는 다음 주부터 확실하게 고치고 싶다.
- **Try** 다음 주에 시도할 것: 서버 인프라 장애나 돌발 상황에서도 학습 리듬이 깨지지 않도록 로컬 개발 환경 활용법을 상시 구축해두고, 공부 분량을 주말이 아닌 주중에 매일 1.5~2시간씩 엄격히 분산해 두어 마감 임박 증후군을 예방하기.
