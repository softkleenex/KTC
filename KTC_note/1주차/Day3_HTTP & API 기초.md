네트워크란

두대 이상의 컴퓨터를 연결하여 네이터를 공유할 수 있도록 연결된 통신시스템

네트워크의 기본요소

1. 클라이언트
	서비스를 요청하는 컴퓨터 또는 사용자
2. 서버
	클라이언트의 요청에 응답하여 서비스를 제공하는 컴퓨터
3. 통신 프로토콜
	네트워크 상에서 데이터를 교환하기 위한 규칙과 절차


IP주소

데이터 패킷의 발신지와 목적지를 식별
네트워크 통신을 가능하게 함

4개의 옷텟(8비트)로 구성, 각 옥텟은 0 -- 255 범위의 숫자, 약 43억개의 고유 주소 제공 가능

IP주소는 다시 네트워크 ID(네트워크 자체를 식별) 와 호스트 ID로 구분가능(네트워크 안의 개별 장치 식별)

그래서, 128 비트(IPv6)체계로 넘어갔다. 자동 구성 및 주소 중복 방지, 향상된 보안기능이 강점

DNS

이를 보완하기 위해 등장한것이 기억하기 좋은 도메인 이름을 쓴다!(52.231.38.95 > www.elice.io)

도메인 이름을 입력하면 DNS서버가 실제 IP주소로 반환한다.

중복 불가능

네트워크의 종류

작은 범주의 네트워크부터

PAN(Personal Area Network)
개이느이 장치들간의 연결을 위한 소규모 네트워크(스마트폰, 무선이어폰 등)

LAN(Local Area Network)
제한된 지역 내의 컴퓨터와 장치들을 연결한 네트워크(사무실, 학교, 가정 내 PC등)


MAN(Metropolitan Area Network)
도시나 대도시 지역을 커버하는 네트워크(도시 내 공공와이파이, 캠퍼스 네트워크)

WAN(Wide Area Network)
넓은 지리적 범위를 커버하는 네트워크(인터넷, 글로벌 기업 네트워크)

네트워크 통신방식

네트워크에서 데이터를 전송하는방법
데이터 전송의 효율성, 신뢰성, 속도 등에 영향을 미친다.

1. 유니 캐스트
	-  한 송신자와 수신자가 1:!로 통신함
	- 네트워크 리소스 효율적 사용
	- 신뢰성 높은 통신방식
		- 웹 브라우징(HTTP, HTTPS), 이메일 전송, 파일 전송 프로토콜)FTP
2. 멀티캐스트
	- 한 송신자가 여러 수신자에게 1:N으로 통신하는 방신
	- 특정 그룹의 수신자에게만 데이터 전송
	- 네트워크 자원 절약 및 효율적인 데이터 분배
		- 라이브 스트리밍, 온라인 강의 네트워크 게임 등에 활용된다
3. 브로드 캐스트
	- 한 송신자가 네트워크 내 모든 대상과 통신하는 방식
	- 네크워크 내 다른 장치들을 발견하고 서비스 제공에 용이함
	- 네트워크 트래픽 증가 가능성있음
		- ARP요청(**"IP 주소를 MAC 주소로 바꿔주는 통역사(또는 명부)"** 역할을 하는 프로토콜입니다.), 네트워크 디스크버리 등





TCP/IP 프로토콜

이전에,, IP프로토콜이 가지는 한계에 대해서 알아보자

IP프로토콜이란? 데이터를 목적지까지 전달하는데 사용되는 핵심 프로토콜, IP주소를 통해 송수신자를 식별해 데이터를 패킷 단위로 전송함
한계는? 신뢰성의 문제가 있다.

핵심 발생 지점 : 단순 데이터를 전달하기만 하고, 전달 과정에서ㄴ의 패킷 손실, 순서뒤바뀜에 책임을 지지않음
	데이터 패킷이 독립적으로 전송되어 순서가 보장되지않음
	패킷 손실시 자동 재전송 혹은 복구 기능이없음

TCP/IP프로토콜은 데이터가 정확한 목적지에 도달할수 있도록 하는 통신규약이다. 인터넷 및 네으워크 통신의 기본 프로토콜 스택(IP 에 TCP를 컬합)

**📌 핵심 설명**
* **전통적인 OSI 7계층 모델**을 실무 환경에 맞게 압축한 **실용적인 구조가 TCP/IP 4계층 모델**입니다.
* **계층 간 대응 관계:**
  * OSI의 **5, 6, 7계층** ➡️ TCP/IP의 **4계층(응용 계층)** 하나로 통합
  * OSI의 **4계층** ➡️ TCP/IP의 **3계층(전송 계층)**
  * OSI의 **3계층** ➡️ TCP/IP의 **2계층(인터넷 계층)**
  * OSI의 **1, 2계층** ➡️ TCP/IP의 **1계층(네트워크 액세스 계층)** 하나로 통합



TCP와 UDP

TCP의 특징
- 연결 지향
	- 3-way handshaking 과 4-way handshaking (통신 전에 연결하는것, 통신 후에 연결을 끊는것)
- 흐름 및 혼잡 제어, 순서를 보장한다
- 높은 신뢰성을 보장한다
- UDP보다 느린 속도
- 이메일 및 파일 전송, 웹페이지 로딩에 쓰임

3way handshaking

user이 serverdptp SVN 메세지를 보내 연결요청
서버는 SYN/ACK 를 동시에 보내 연결 수락
user은 server에세 ACK를 보내 연결 성공


4way handshaking

user이 server에세 fin을 보냄
server에서 user에게 ack보냄
	fin보냄
user이 server에게 ack 보냄

UDP의 특징

- 비 연결형 지향(3way, 4way handshaking 같은거 없음)
- 낮은 신뢰성, 순서 보장 X
- 정보의 신호 절차를 거치지않는다
- 스트리밍, 온라인 게임 등 실시간 서비스에 적합하다.


|       구분        |   TCP    |      UDP      |
| :-------------: | :------: | :-----------: |
|    **연결 방식**    |   연결형    |     비연결형      |
|    **전송 순서**    | 전송 순서 보장 |  전송 순서 보장 X   |
|  **수신 여부 확인**   |    O     |       X       |
|    **통신 방식**    |   1:1    | 1:1, 1:N, N:N |
|     **신뢰성**     |    높음    |      낮음       |
| **오류 검출 및 재전송** |    O     |       X       |
|     **속도**      |    느림    |      빠름       |

HTTPS 프로토콜과 RESUFUL

브라우저는 어떻게 작동할까?

1. url을 입력하면 dns서버에서 해당 도메인을 검색하여 ip주소를 반환한다.

2. 브라우저 http 요청 메세지가 생성된다
		이 단계는 어느계층일까? 사용자가 입력에 따라 데이터가 생성되는, 가장 위의 응용계층에서 작동한다.
3. TCP/IP패킷 생성
		이 데이터를 패킷으로, 이 패킷을 프레임으로 포장하여 서버에게 잘 도착하게한다


TTP 메시지가 **TCP 세그먼트로 1차 포장되고, 그게 다시 IP 패킷에 싸인 뒤**, 최종적으로 **프레임(Frame)으로 캡슐화되어** 랜선(물리 계층)을 타고 서버에게 보내진다!

4. 이후 이 프레임이 서버가 있는곳으로 도착하면, 이 프레임을 패킷으로, 이 패킷을 데이터로 해체하여 내용을 확인한다.(역 캡슐화)
5. 이제 이 응답에 대해서 서버가 대답해야한다. 문제 여부와 상관없이 항당 사용자에게 메시지로 응답해야한다.
	- 예를들면, https, json, javascript 등으로 응답해야한다.
6. 이 과정 이후 최종적으로 화면을 사용자에게 보여주게된다.


HTTP프로토콜과 핵심 특징

프로토콜 : 통신의 과정에 필요한 형식, 약속, 규칙 등을 정의한것
- 네트워크 통신에서 데이터를 교환하기 위한 규칙과 표준 제공
- 데이터 전송의 일관성 보장
- 다양한 기기와 시스템 간의 상호 운용성 제공
- 오류 검출 및 수정기능 제공

프로토콜의 기능


| 기능 | 역할 |
| :--- | :--- |
| **단편화/재합성** | 데이터 블록을 나누고 합쳐서 전송 |

왜 HTTP프로토콜이 중요한걸까?

HTTP : HyperText Transfer Protocol
클라이언트 - 서버 간 데이터가 교환되는 방식에 대한 규약
거의 모든 형태의 데이터 전송 가능
무상태성과 비연결성


무상태성
- 서버가 클라이언트의 상태를 보존하지않음
- 각 요청은 독립적이고, 이전요청과의 연결성이없음
- 서버의 확장성에 용이

이게 왜 좋은가?
- 여러 서버중에 아무에게나 요청 가능
- 설계가 단순하고, 수평적 확장성을 높일 수 있다
- 요청이 많아지면 그냥 점원의 수를 늘려 문제해결가능



비연결성
- 클라이언트 - 서버 간연결을 유지하지않음(응답과 동시에 연결이 지속되지않고 종료됨)
- 서버의 불필요한 연결을 줄이고 많은 클라이언트의 요청을 동시에 처리 가능

이게 왜 좋은가?
- 연결이 만약 지속된다면 다른 요청은 응답이 어렵다(사버 자원의 효율화에 유리함)
- 서버의 불필요한 연결을 줄이고 많은 클라이언트의 요청을 통시 처리 가능.

HTTP프로토콜의 연장인 HTTPS(HTTP + S-secure)

HTTPS는 HTTP의 보안버전이다
데이터 암호화로 데이터 보호 및 신뢰성을 향상한다
TLS/SSL 프로토콜을 사용한다.

HTTP 메서드

클라이언트가 서버에게 수행할 작업의 종류를 지시하는 명령어
RESTful API설계에서 중요한 역할을 한다.


CRUD
	Create > Post(자료를 등록할 떄 사용)
	Read > Get(자료를 조회할 떄 사용)
	Update > Put/Patch(자료를 수정할 떄 사용)
	Delete > Delete(자료를 삭제할 떄 사용)
4가지 기본 기능의 약어, HTTP메서드의 4가지 기능과 대응됨

GET 메서드
- 서버에서 리소스 요청, 데이터 조회
- 데이터 검색, 웹페이지 로딩 등에 사용
- 요청 본문 없음
- 요청 URL에 쿼리 파라미터를 포함할 수 있다.

POST 메서드
- 서버에 데이터를 전송하여 새로운 데이터를 생성
- 폼 제출, 파일 업로드, 회원 가입 등에 사용
- 요청 본문에 데이터 포함
- 서버의 상태를 변경함

PUT 메서드
- 서버에 데이터를 전송하여 리소스를 완전히 대체함
	- PATCH메서드는 일부 대체함
- 데이터 전체 갱신시에 사용
- 요청 본문에 데이터 포함됨

Delete 메서드
- 서버에서 특정 리소스를 삭제함
- 요청 본문 없음(GET 메서드와 동일이유, 리소스를 삭제하는데이 특정 추가 데이터가 필요없기에)


그 외 메서드들

HEAD 메서드 GET 요청과 유사, 응답 본문 포함 X
OPTIONS 메서드 서버가 지원하는 HTTP 메서드 목록 요청
TRACE 메서드 리소스에 대한 경로를 다라 메시지 루프백 메시지 수행

### HTTP 주요 요청 헤더 (Request Headers)

| 헤더 필드 | 설명 |
| :--- | :--- |
| **`Host`** | 요청하는 호스트의 이름 및 포트 번호 |
| **`User-agent`** | 클라이언트가 사용하는 웹 브라우저 정보 (예: 크롬, 사파리 등) |
| **`Referer`** | 현재 요청된 페이지로 이동하기 직전에 방문한 웹 페이지의 URL |
| **`Accept`** | 클라이언트가 처리할 수 있는 데이터 타입 및 우선순위 |
| **`If-Modified-Since`** | 값으로 지정한 날짜 이후에 수정된 자원 요청 (캐시 활용 시 사용) |
| **`Authorization`** | 클라이언트가 가지고 있는 서버에 대한 권한(인증) 정보 |
| **`Origin`** | 요청이 시작된 주소 정보 (CORS 방어 등에서 중요하게 쓰임) |
| **`Cookie`** | 서버에 의해 설정된 클라이언트의 쿠키 정보 (로그인 상태 유지 등) |

### HTTP 주요 응답 헤더 (Response Headers)

| 헤더 필드 | 설명 |
| :--- | :--- |
| **`Location`** | 리다이렉션 해야 하는 페이지의 URL |
| **`Server`** | 서버에서 사용하는 소프트웨어 정보 (예: Apache, Nginx) |
| **`Age`** | 응답이 생성되고 지난 초 단위의 시간 |
| **`Referrer-Policy`** | 요청 메시지에 포함할 Referer 정보의 양을 제어 |
| **`WWW-Authenticate`** | 클라이언트가 서버에 인증해야 하는 방식 요구 (보통 401 에러와 함께 옴) |
| **`Proxy-Authenticate`** | 클라이언트가 프록시 서버로 인증 정보를 보내는 방식 요구 |



HTTP 상태 코드

HTTP 상태코드 : 클라이언트의 요청에 대한 서버의 응답을 나타내는 3자리 숫자코드
### HTTP 상태 코드 (Status Code) 요약

| 코드 대분류 | 의미 | 설명 |
| :--- | :--- | :--- |
| **1xx** | 정보 응답 (Informational) | 요청을 받아들여 처리 중임을 나타냄 |
| **2xx** | 성공 (Successful) | 요청이 성공적으로 접수되고 처리됨 |
| **3xx** | 리다이렉션 (Redirection) | 요청 완료를 위해 클라이언트의 추가 작업(다른 주소로 이동 등)이 필요함 |
| **4xx** | 클라이언트 오류 (Client Error) | 클라이언트의 요청 문법이 잘못되었거나, 요청을 처리할 수 없음 |
| **5xx** | 서버 오류 (Server Error) | 서버 측의 문제로 인해 정상적인 요청을 처리하지 못함 |

각 카테고리의 대표 넘버


### HTTP 상태 코드: 1xx (정보 응답)
**"요청을 받았으며 프로세스를 계속함"**

| 코드 | 설명 |
| :--- | :--- |
| **100** | **Continue**<br>- 클라이언트가 요청의 첫 번째 부분을 전송했으며, 서버가 나머지 요청을 계속해서 전송하라고 응답 |
| **101** | **Switching Protocol**<br>- 클라이언트가 프로토콜 전환을 요청했으며, 서버가 이를 승인 |
| **102** | **Processing**<br>- 서버가 요청을 수신하여 처리 중이지만, 아직 완료되지 않았음을 나타냄 |
| **103** | **Checkpoint**<br>- 응답 헤더의 초기 힌트를 클라이언트에게 제공하여 리소스 로드를 가속화 |

### HTTP 상태 코드: 2xx (성공)
**"요청을 성공적으로 수신하고 실행함"**

| 코드 | 설명 |
| :--- | :--- |
| **200** | **OK**<br>- 요청이 성공적으로 처리되었음을 나타냄 |
| **201** | **Created**<br>- 요청이 성공적으로 처리되었으며, 새로운 리소스가 생성되었음을 나타냄 |
| **202** | **Accepted**<br>- 요청이 수락되었지만, 아직 처리되지 않았음을 나타냄 |
| **203** | **Non-Authoritative Information**<br>- 요청이 성공적으로 처리되었지만, 응답 헤더 정보가 원본 서버가 아닌 다른 소스로부터 왔음을 나타냄 |
| **204** | **No Content**<br>- 요청이 성공적으로 처리되었지만, 반환할 내용이 없음을 나타냄 |


### HTTP 상태 코드: 3xx (리다이렉션)
**"요청 완료를 위해 추가 작업 조치 필요"**

| 코드 | 설명 |
| :--- | :--- |
| **300** | **Multiple Choices**<br>- 요청한 리소스에 대한 여러 가지 선택지가 있음을 나타냄 |
| **301** | **Moved Permanently**<br>- 요청한 리소스가 영구적으로 새로운 URL로 이동되었음을 나타냄 |
| **302** | **Found**<br>- 요청한 리소스가 임시로 다른 URL에 위치함을 나타냄 |
| **304** | **Not Modified**<br>- 요청한 리소스가 수정되지 않았으며, 클라이언트의 캐시된 버전을 사용할 수 있음을 나타냄 |

### HTTP 상태 코드: 4xx (클라이언트 오류)
**"클라이언트의 요청 오류"**

| 코드      | 설명                                                     |
| :------ | :----------------------------------------------------- |
| **400** | **Bad Request**<br>- 클라이언트의 요청이 잘못되었음을 나타냄             |
| **401** | **Unauthorized**<br>- 인증이 필요하지만 제공되지 않았거나 유효하지 않음을 나타냄 |
| **403** | **Forbidden**<br>- 서버가 요청을 이해했지만, 권한이 없어서 요청을 거부함      |
| **404** | **Not Found**<br>- 요청한 리소스를 찾을 수 없음을 나타냄               |
| **405** | **Method Not Allowed**<br>- 요청한 메서드가 허용되지 않음을 나타냄      |

### HTTP 상태 코드: 5xx (서버 오류)
**"서버의 응답 오류"**

| 코드      | 설명                                                                  |
| :------ | :------------------------------------------------------------------ |
| **500** | **Internal Server Error**<br>- 서버가 요청을 처리하는 도중 예기치 않은 상태에 도달했음을 나타냄 |
| **501** | **Not Implemented**<br>- 서버가 요청을 수행할 방법을 지원하지 않음을 나타냄               |
| **502** | **Bad Gateway**<br>- 게이트웨이 또는 프록시 서버가 상위 서버로부터 잘못된 응답을 받았음을 나타냄     |
| **503** | **Service Unavailable**<br>- 서버가 일시적으로 요청을 처리할 수 없음을 나타냄            |


REST API 개요

REST
	Representational State Transfer
	Http 프로토콜에 기반하여 자원을 관리하는 방식을 정의
	자원을 이름으로 식별하여 자원의 상태 정보를 주고 받음
API
	Application Programing Interface
	시스템 간 상호작용을 위한 인터페이스
	레스토랑의 점원


즉, REST API란


REST의 원리(HTTP 프로토콜)을 따르는 API이다

RESTAPI의 특징

- 클라이언트 - 서버(Client-Server) 구조
	- 사용자 인터페이스와 비즈니스 로직/데이터 저장의 분리
- 무상태(Stateless)
	- 각 요청은 독립적, 서버에 상태 정보 유지X
- 캐시 처리 가능
	- 응답은 캐싱 가능해야함
- 계층화 시스템
	- 클라이언트는 서버를 직접 호출하지않음 -> 다중 계층 허용(로드밸러ㄴ싱, 보안, 암호화 등)
- 인터페이스 일관성
	- 효율적 아키텍처를 위한 일관된 인터페이스
- 코드 온 디멘드
	- 서버는 실행 가능 코드 형태로 기능을 전홍할 수 있음


이런 원칙을 잘 준수한 API를 RESTFUL API라고 한다!

REST API의 구성요소


1. 자원(Resources) - URL
	 모든 자원에 고유한 id가 존재
	 데이터 베이스의 엔티티나 정보의 단위
		 사용자, 제품, 주문 등
		 

URI란? Uniform Resource Identifier
URI는 정보의 자원을 표현함, 즉 지원식별자이다.


URI의 구조

### URL(URI) 구조 분석
**예시:** `http://www.example.com:8080/resource/page.html?param1=value1&param2=value2#section1`

| 구성 요소 | 예시 부분 | 설명 |
| :--- | :--- | :--- |
| **Scheme** | `http` | 리소스에 접근하기 위한 프로토콜 (웹은 주로 http, https) |
| **Host** | `www.example.com` | 리소스를 호스팅하는 서버 이름(도메인) 또는 IP 주소 |
| **Port** | `8080` | 서버에서 리소스에 접근하기 위해 사용되는 게이트 (웹 기본값: HTTP 80, HTTPS 443) |
| **Path** | `/resource/page.html` | 호스트 내에서 리소스의 정확한 위치 (폴더/파일 경로) |
| **Query Parameters** | `?param1=value1...` | 리소스에 전달되는 추가 정보 (주로 검색어나 필터링 옵션) |
| **Fragment Identifier** | `#section1` | 부가적인 식별자 (해당 웹 페이지 내의 특정 스크롤 위치 지정) |


2. 행위(Verbs) - HTTP 메서드
	자원에 대한 조작을 정의함



| 메서드 (Method + URL) | 역할 |
| :--- | :--- |
| **[GET]** `/users/{userId}` | 특정 사용자의 상세 정보를 **조회** (Read) |
| **[POST]** `/users` | 특정 사용자를 **등록** (Create) |
| **[PUT]** `/users/{userId}` | 특정 사용자의 정보를 **수정** (Update) |
| **[DELETE]** `/users/{userId}` | 특정 사용자의 정보를 **삭제** (Delete) |

이것들은 모두 일관적이여야한다!

> [!info]- RESTful API URL 의 일관성
> 1. **리소스명은 동사보다는 명사를 사용한다.**
> 2. **리소스에 대한 행위는 HTTP 메서드로 표현한다.**
> 3. **URL 경로에는 소문자를 사용한다.**
> 4. **계층 관계는 슬래시(`/`)로 구분한다.**
> 5. **마지막 문자로 슬래시(`/`)를 사용하지 않는다.**
> 6. **의미 구분을 위해서는 하이픈(`-`)을 사용하고 언더바(`_`)를 사용하지 않는다.**



3. 표현(Representations) - HTTP 메시지 페이로드
	클라이언트가 자원의 상태에 대한 조작을 요청 -> 서버응답
	자원은 JSON, XML, TEXT 등 여러 형태를 가질 수 있음

JSON이 보통 가장 많이 쓰인다.
경량의 데이터 교환형식, 프로그래밍 언어 독립적.
키/값 쌍의 집합 구조이다.


REST API의 기본 설계 원칙

1. 자원 식별
	인터넷에 있는 모든것은 고유 주소(URI)ㄹㄹ 가져야한다
	자원이 URI로 식별되어야한다, 특정 자원은 오직 하나으 ㅣURL에 매핑되어야한다
2. 표현을 통한 자원의 조작
	웹 페이지의 정보를 조작할 떄 해당 정보는 XML, JSON 같은 형식으로 수성된다
	이같이 특정 형식을 통해 정보에 대한 조회, 수정, 삭제등을 수행해야한다
	URL을 바꾸지 않고도 표현을 통해 리소스를 조작 가능해야한다
3. 자기서술 적 메시지
	메시지가 자기 자신을 어떻게 처리해야할지 설명해야한다
	수신하는 쪽에서 별도의 설명 없이도 해당 메시지를 이해하고 처리할 수 있도록 정의
4. HATEOAS(Hypermedia As The Engine of Application State)
	웹 페이지가 다른 페이지나 상태로 넘어갈 수 있도록 링크를 제공해야한다
	링크를 통해 사용자나 프로그램이 다음에할 수 있는 동작을 유추 가능하다
